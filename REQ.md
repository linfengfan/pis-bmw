## 一、背景与目标

### 1.1 问题痛点

Claude 等 LLM 在读取大型 Markdown 文件时 token 消耗极大（单个文件 200KB+，每次读取 50,000+ token），导致：
- 对话成本高，响应慢
- 无法精准定位文件中某个数据点
- 上下文窗口被大文件污染，核心信息被稀释

### 1.2 解决思路

将文档**预先向量化**，存入本地向量数据库。查询时先用语义检索找出最相关的文本块（~3-5KB），再将精准片段喂给 LLM，实现**按需读取、精准回答**。

Token 消耗从每次 ~50,000 → ~500，节省 **99%**。

### 1.3 核心目标

| 目标 | 描述 |
|------|------|
| 降 Token 消耗 | 向量化后每次查询仅消耗 ~500 token |
| 多格式支持 | PDF / Markdown / 网页 URL |
| 本地化存储 | 所有数据存在本地，不上云 |
| 零运维 | 无数据库服务器进程，本地文件即数据库 |
| 可扩展 | 支持后续增加 PDF 解析精度、OCR、批量导入 |

## 二、功能需求

### 2.1 功能清单（按优先级）

#### P0 — MVP 必须

- [ ] **文件上传**：支持 PDF、Markdown、TXT 上传，前端拖拽/点击
- [ ] **URL 抓取**：输入 URL → 自动爬取 → HTML 转 Markdown → 向量化
- [ ] **语义检索**：输入自然语言问题 → 返回最相关的文本块（Top-K 可调）
- [ ] **向量库持久化**：ChromaDB 数据存储在本地，重启后不丢失
- [ ] **文档列表**：查看已入库的所有文档，支持删除

#### P1 — V1 扩展

- [ ] **批量上传**：一次上传多个文件
- [ ] **文档元信息**：文件名、来源 URL、上传时间、块数量
- [ ] **搜索高亮**：检索结果中标注关键词
- [ ] **PDF 元信息提取**：标题、作者、页数

#### P2 — 未来规划

- [ ] **OCR 支持**：扫描版 PDF（图片型）自动识别文字
- [ ] **多语言 Embedding**：英文报告（World Silver Survey 等）用英文模型
- [ ] **LLM 整合**：直接提问 → 检索 → LLM 生成答案（Agentic 模式）
- [ ] **文档分类**：年报、公告、研报自动打标签
- [ ] **版本管理**：同一文件多次上传时自动覆盖或保留历史
- [ ] **导出结果**：将检索结果导出为 Markdown / PDF
- [ ] **API Key 管理**：接入 Claude/GPT API 作为 LLM 层
- [ ] **增量更新**：文档更新时只重新向量化变化的部分

## 三、数据规格

### 3.1 输入数据格式

| 类型 | 格式 | 最大尺寸 |
|------|------|---------|
| PDF | .pdf | 50MB |
| Markdown | .md / .markdown | 无限制 |
| 纯文本 | .txt | 无限制 |
| 网页 | URL（http/https） | 无限制 |

### 3.2 向量化参数

| 参数 | 当前值 | 说明 |
|------|--------|------|
| Embedding 模型 | `shibing624/text2vec-base-chinese` | 中文语义，免费离线 |
| 分块大小 | 800 字/块 | 保留上下文的最小单位 |
| 段间重叠 | 100 字 | 避免块边界切断语义 |
| 最少块字数 | 50 字 | 过滤噪音块 |
| Top-K 默认值 | 5 | 可调范围 1-20 |

### 3.3 向量数据库结构

```
Collection: docs

每条记录字段：
  - id:         "{doc_id}_{chunk_index}"  (唯一)
  - document:  "文本内容"                  (str)
  - embedding: [0.123, ...]               (768维 float)
  - metadata:
      - doc_id:       str
      - filename:     str
      - title:        str
      - source_type:  "file" | "url"
      - created_at:   ISO8601
      - url:          str (仅 url 来源)
      - chunk_index:  int
      - chunk_total:  int
```

### 3.4 存储路径

```
data/
├── raw/           # 原始上传文件（按 doc_id 命名）
├── md/            # URL 转换后的 Markdown 文件
└── chroma_db/     # ChromaDB SQLite 数据库文件
```

## 四、非功能需求

| 类型 | 要求 |
|------|------|
| **性能** | 单文件向量化 < 30s（PDF < 50MB） |
| **可维护性** | 模块化架构，新增解析器不影响其他模块 |
| **可移植性** | Python 3.10+ / Node 18+，无平台依赖 |
| **数据安全** | 所有数据本地，不上传任何第三方 |
| **错误处理** | 上传失败/解析失败/向量化失败 均返回明确错误信息 |
| **降级策略** | Playwright 渲染失败 → 回退到 requests + BeautifulSoup |
