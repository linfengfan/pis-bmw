# 规范即文档原则 (Specs as Living Docs)

## 核心定义
`openspec/specs/` 目录作为永远与代码保持同步的活体文档。每次变更归档后自动合并规范增量到 specs/。

## 实施方式

### 目录结构
```
openspec/
└── specs/
    ├── auth/
    │   ├── login-flow.md
    │   └── token-management.md
    ├── api/
    │   └── rest-conventions.md
    └── ai-pipeline/
        └── agent-orchestration.md
```

### 同步机制
| 触发事件 | 动作 |
|----------|------|
| 新功能上线 | 将 proposal + design 摘要合并至对应 spec |
| 重大重构 | 更新 spec 架构图 + 决策记录 |
| BUG 修复 | 在 spec 中补充"已知约束"章节 |

### 禁止事项
- ❌ 绕过流程直接手写核心业务逻辑
- ❌ spec 与代码不一致（代码即事实，spec 落后则删除/更新 spec）
- ❌ spec 包含已废弃的 API 或路径

## 为什么重要
- 传统"先写文档再写代码" → 文档必然过期
- "规范即文档" → 代码变更驱动规范更新，永远保鲜
- 多人协作时，spec 是唯一可信的事实来源

## 触发条件
任何涉及跨模块/跨团队的系统性变更。
