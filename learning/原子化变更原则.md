# 原子化变更原则

## 核心定义
每次变更只解决一个需求。避免上下文过长导致 LLM 遗忘或幻觉。

## 三条铁律

### 1. 单一职责（一次一件事）
一个 PR / 一个 commit / 一次 AI 对话只做一件事：
- ❌ "重构 Auth 模块 + 修复 BUG + 添加测试"
- ✅ "重构 Auth 模块"（其他两个分开处理）

### 2. 幂等性强制
所有自动化操作必须可重入、不产生副作用：
- 同一命令执行两次 = 同一结果
- 禁止：创建不可重复资源（不指定唯一 ID）、手动修改不可回滚状态
- 允许：幂等的文件写入（覆盖）、数据库幂等更新

### 3. 小步快跑
保持人工 Review 在可控范围内：
- 单次变更 ≤ 300 行核心逻辑
- 超过则拆分
- 优先合并"能跑的骨架"，再迭代细节

## 为什么重要
- 上下文窗口有限 → 大段变更导致 LLM 在后半段"失忆"
- 原子化变更 → Reviewer 容易定位问题，CI 容易定位故障
- 幂等性 → 重复执行不翻车，降低操作事故率

## 触发条件
任何代码变更场景均默认遵守。
